# N225LLMAdvisor システム改善判断指示書

**バージョン**: 1.0
**作成日**: 2026-04-23
**対象**: N225LLMAdvisor プロジェクト運用中の改善判断
**目的**: 運用データから「どこを改善すべきか」を客観的に判断するための基準

---

## 0. この指示書の使い方

### いつ使うか

以下のタイミングで、このドキュメントを開いてチェックしてください：

1. **週次チェック**: 毎週土曜または日曜（1週間 = 5営業日分のデータを振り返る）
2. **月次チェック**: 毎月末（1ヶ月 = 約20営業日分のデータを振り返る）
3. **違和感を感じた時**: 「何かおかしい」と感じた時

### 何をするか

このドキュメントの各セクションは**チェックリスト形式**です。該当する項目にチェックを入れ、最後に「改善すべきポイント」を特定してください。

**重要**: チェックした項目が多いほど優先度が高いわけではありません。**1つでも「致命的」に該当すれば即座に対応**する項目があります。

---

## 1. 運用継続可否の判断（致命的問題チェック）

**このセクションに1つでも該当する項目があれば、運用の一時停止を検討してください。**

### 1.1 記録の継続性

- [ ] 直近2週間で、記録できた日が50%未満（5営業日中2日以下）
- [ ] 朝の分析 or 結果記録 or 振り返りのどれかを、3日以上連続でスキップした
- [ ] 「記録が面倒」と感じる頻度が増えている

**該当した場合**:
- 記録項目を**さらに削減**してください（最重要3項目のみに絞る）
- または**運用時間を固定**してください（朝は必ず7:30〜8:00 の30分、夜は必ず15:50〜16:10 の20分）
- それでも続かない場合、プロジェクト全体の見直しが必要

### 1.2 MCPサーバーの安定性

- [ ] 週1回以上、MCP経由のスクリーンショット取得でエラーが発生している
- [ ] `pine_switch_script` や `chart_download_csv` で失敗が発生している（旧 `pine_open` は使用禁止）
- [ ] 記録時に「MCPエラーで待たされる」時間が合計10分/日を超える

**該当した場合**:
- 既存プロジェクト側の MCP 安定化（H1〜H4 の対応）を最優先する
- LLM Advisor の運用は一旦停止し、MCP 安定後に再開

### 1.3 データ整合性

- [ ] YAML パースエラーが週2回以上発生している
- [ ] `day_high`, `day_low` などの基本数値が取得できない日がある
- [ ] シナリオの `fired: true/false` の判定で迷うケースが週3回以上ある

**該当した場合**:
- テンプレートの該当フィールドを**より機械判定可能な形式**に修正する
- 詳細は「3. テンプレート改善判断」セクションを参照

## 2. データ蓄積状況の評価

### 2.1 最小データ量の確認（最初の30営業日）

以下の数値が揃うまで、改善よりも**データ蓄積を優先**してください：

- 記録完了営業日数: **最低30日**
- 朝の分析 + 結果記録 + 振り返りが全て揃った日: **最低20日**
- シナリオが発動した日: **最低10日**

**30日未満の場合の判断**:
- 改善アクションは最小限に留める
- 「致命的問題」のみ対応
- それ以外はデータが30日貯まるまで記録を続ける

**理由**: データ量が少ないうちに改善すると、偶然の偏りに反応して戦略を歪めます。

### 2.2 データ品質のチェック

30日以上のデータが貯まったら、以下を確認：

- [ ] scenarios の `probability` 合計が100%になっている日が80%以上
- [ ] `trigger_level` が実際の価格レンジ内に収まっている（完全に外れている場合は警告）
- [ ] `fired_time` が `time_window` の範囲内に収まっている
- [ ] `winning_scenario` が記録されている日が80%以上

**該当しない項目があれば、3. テンプレート改善へ**

## 3. テンプレート改善判断

### 3.1 daily_analysis_template.md の改善ポイント

以下のいずれかが頻発している場合、テンプレートを改善してください：

#### 3.1.1 シナリオ判定の曖昧さ

- [ ] 「このシナリオは発動したのか判定に迷う」が週3回以上発生
- [ ] `type: break_above` の条件が、実相場の動きと噛み合わない
- [ ] `confirmation` の方法（`close_5m_above` 等）が実際には機能していない

**改善指針**:
```yaml
# 現在の曖昧な構造
entry:
  type: "break_above"
  level: 0
  confirmation: "close_5m_above"

# 改善後（より厳密に）
entry:
  type: "break_above"
  level: 0
  confirmation:
    method: "close_5m_above"
    consecutive_bars: 1        # 何本連続で上抜けが必要か
    min_pts_above: 5           # 最小上抜け幅
  volume_check:
    required: false            # 出来高確認の要否
    min_ratio: 1.0             # 最小出来高倍率
```

#### 3.1.2 シナリオの網羅性不足

- [ ] 「A/B/C のどれにも当てはまらない動き」が週2回以上発生
- [ ] 「寄り天」「ギャップダウン継続」など**特殊展開**が記録されていない

**改善指針**: scenarios を 3 → 5 に拡張：
- A: 上昇継続
- B: レンジ
- C: 下落継続
- **D: 寄り天・高値圏反転**（新規追加）
- **E: ギャップダウン継続**（新規追加）

ただし、最初から5シナリオにしない。**発生頻度が高い特殊展開**のみ追加する。

#### 3.1.3 確率の扱い

- [ ] probability の合計が100%にならない日が多い
- [ ] probability が毎日 40/35/25 などの**決まった比率**になっている（Claude の癖）

**改善指針**:
```yaml
# YAML 側で制約を明記
scenarios_constraint:
  probability_sum_must_equal: 1.0
  probability_min_per_scenario: 0.05   # どのシナリオも最低5%
  probability_max_per_scenario: 0.80   # どれか1つに集中させない
```

### 3.2 result_template.md の改善ポイント

#### 3.2.1 予測評価の主観性

- [ ] `prediction_accuracy: high/medium/low` の判定が自分でもブレる
- [ ] 「high と書いたが後から見ると medium だった」が多発

**改善指針**: 主観評価を廃止し、**自動計算される指標**に置換：

```yaml
prediction_evaluation:
  direction_match: true/false      # 朝予測した主方向と実際の終値方向が一致したか
  range_containment: true/false    # 実際の高安が予測レンジ内に収まったか
  top_scenario_fired: true/false   # 最高確率のシナリオが発動したか
  brier_score: 0.15                # 自動計算値（低いほど良い）
```

#### 3.2.2 未発動日の扱い

- [ ] ノートレード日が週2日以上発生
- [ ] 「シナリオ予測は合っていたがトレードしなかった」日の評価が曖昧

**改善指針**:
```yaml
no_trade_analysis:
  reason: "all_scenarios_invalidated" | "out_of_time_window" | "waited_too_long" | "manual_decision"
  would_have_been_profitable: true/false  # 仮にトレードしていたら利益だったか
  regret_level: "high/medium/low/none"    # 機会損失の実感
```

### 3.3 review_template.md の改善ポイント

#### 3.3.1 Q2「乖離の理由」の改善

- [ ] 「ノイズ/運の影響」と書く頻度が月5回以上
- [ ] Q2 の記述が毎回似たような内容になっている

**改善指針**: Q2 を構造化する：

```markdown
## Q2: 予測と実績が乖離した理由（構造化）

### 乖離のタイプ（チェック）
- [ ] 方向性の誤り（up と予想したが down だった）
- [ ] 強度の誤り（小幅と予想したが大幅だった、またはその逆）
- [ ] タイミングの誤り（時間帯の読み違い）
- [ ] イベントの影響（予期せぬニュース）
- [ ] パターン認識の誤り（類似パターンと違う展開）

### 分析可能な要因
- [ ] 前日夜の米国市場の影響を過小評価
- [ ] テクニカル水準の見落とし（具体的な価格を記載）
- [ ] ボラティリティレジームの変化を認識できなかった
- [ ] 相関関係（NQ/原油/ドル円）の変化を捉えられなかった

### 分析困難な要因（ノイズ）
- [ ] 単発ニュース（原因追跡可能）
- [ ] 謎の動き（原因追跡不能）
```

これにより、月末に集計して「乖離のタイプ別頻度」が見えるようになります。

#### 3.3.2 Q5「累積すべき学び」の昇格ルール

現在は「5回以上で high」ですが、サンプル数による偏りがあります。

**改善指針**: ベイズ的な信頼区間ベースの判定：

```yaml
# memory/lessons.md の各 lesson に追加
lessons:
  - id: "L001"
    pattern: "Range 局面では NQ と日経の相関が弱まる"
    observations: 10
    confirmations: 7   # 観察のうち一致した回数
    confidence_score: 0.70  # confirmations / observations
    confidence_level: "medium"  # 以下の基準で自動判定

# 信頼度の新ルール
confidence_rules:
  high: "observations >= 10 AND confidence_score >= 0.70"
  medium: "observations >= 5 AND confidence_score >= 0.60"
  low: "観察回数に関わらず、上記以外"
```

## 4. シナリオ生成プロンプトの改善判断

### 4.1 Claude のシナリオ生成が同質化している兆候

- [ ] 30日のうち、同じような構成（上昇/レンジ/下落 = 40/35/25）の日が80%以上
- [ ] 「VWAP」「前日高値」「サポレジ」という単語が毎日登場する
- [ ] `thesis` の表現が毎回似ている

**改善指針**: `/analyze` のシステムプロンプトに**多様性確保のルール**を追加：

```
シナリオ生成時の追加ルール:
1. 過去7日間の daily_analysis を参照し、同じような構成のシナリオが
   連続しないようにする
2. 毎回必ず「逆説的なシナリオ」を1つ含める
   （市場コンセンサスに反する展開）
3. `thesis` は毎回異なる切り口で書く
   （テクニカル / ファンダメンタル / 相関 / 時間帯特性 / 季節性）
4. probability の分布を1つのパターンに固定しない
   （時には 60/30/10 のように極端に偏る予測も可）
```

### 4.2 予測精度が偶然と区別できない

- [ ] 30日データで `direction_match` の正解率が45〜55%（コイン投げと同等）
- [ ] Brier Score が 0.25 以上（未予測と同じレベル）

**該当した場合の判断**:
1. まずデータを60日まで増やして再評価（偶然の可能性）
2. それでも改善しない場合、**シナリオ生成プロンプトを根本から見直し**
3. それでも改善しない場合、**LLM 駆動アプローチの見直し**（この場合は重大な判断）

## 5. 既存プロジェクトとの役割分担の判断

### 5.1 LLM Advisor が既存戦略を補完できているかの判断

以下のどれかが明確になってきたら、統合の検討を始める：

- [ ] 「朝の分析で〇〇と判断した日は、既存戦略 L1 スキャルプの勝率が高い」という相関が見つかる
- [ ] 「朝の分析で△△と判断した日は、既存戦略を停止した方が良い」という相関が見つかる
- [ ] LLM 単独の予測が、既存戦略の勝率より一貫して高い

**該当した場合の次ステップ**:
- CLAUDE.md に「既存戦略と LLM Advisor の統合ルール」を追加
- パターン1（ON/OFF フィルター）、パターン2（独立並走）、パターン3（パラメータ調整）のどれを採用するか決定

### 5.2 LLM Advisor が既存戦略と無関係な場合

- [ ] 60日経過しても、既存戦略との相関が見つからない
- [ ] LLM 予測の精度が低い（前章の改善指針を実施済み）

**該当した場合の判断**:
- LLM Advisor は「データ分析・相場観の蓄積」の役割に限定
- 執行判断への統合は見送り
- Phase 3（Groq 拡張）は延期または中止

## 6. 優先度付きの改善アクション表

運用中に複数の改善点が見つかった時、以下の優先順位で対応してください：

| 優先度 | 改善対象 | トリガー条件 | 対応時期 |
|-------|---------|-------------|---------|
| **最高** | 記録の継続性 | 記録できない日が多い | 即座に対応 |
| **最高** | MCPサーバーエラー | 週1回以上エラー | 即座に対応 |
| **高** | テンプレートのパースエラー | 週2回以上発生 | 1週間以内 |
| **高** | シナリオ判定の曖昧さ | 週3回以上迷う | 2週間以内 |
| **中** | 予測精度指標の主観性 | 自分でブレる自覚 | 30日データ後 |
| **中** | シナリオの網羅性不足 | 特殊展開の頻発 | 30日データ後 |
| **低** | レビュー構造化 | Q2 が定型化 | 60日データ後 |
| **低** | 信頼度ルール変更 | lessons 5件以上 | 60日データ後 |
| **特別** | LLM 駆動の根本見直し | 60日で精度45%以下 | 要慎重判断 |

## 7. 改善判断フローチャート

```
週次チェック開始
    ↓
【1. 致命的問題】に該当？
    Yes → 即座に対応 → 運用再開
    No  ↓
【2. 最低データ量】30日達成？
    No  → データ蓄積継続、改善は最小限
    Yes ↓
【3. データ品質】問題あり？
    Yes → テンプレート改善
    No  ↓
【4. シナリオ同質化】していないか？
    Yes → プロンプト改善
    No  ↓
【5. 予測精度】改善傾向にあるか？
    No  → 60日後に根本見直し判断
    Yes ↓
【6. 既存戦略との相関】見えてきたか？
    Yes → 統合ルール策定
    No  → さらに30日データ蓄積
```

## 8. 改善履歴の記録

改善を実施したら、以下の形式で `docs/improvement_log.md` に記録してください：

```markdown
## 2026-XX-XX 改善実施

### トリガー
- チェックリストのどの項目が該当したか
- 発生頻度・期間

### 実施した改善
- 変更したファイル
- 変更内容（before/after）

### 期待効果
- 何が改善されるか

### 事後評価（実施後2週間）
- 期待通りの効果が出たか
- 副作用はなかったか
- さらなる改善が必要か
```

これを続けることで、**「どの改善が効果的だったか」の履歴**が残ります。

## 9. やってはいけないこと（反面教師）

以下の行動は**プロジェクトを破綻させます**。意識的に避けてください：

### 9.1 データ不足での早期判断

- 10日のデータで「予測精度が低いから LLM は使えない」と判断する
- 20日のデータで「このシナリオタイプは不要」と削除する
- 30日のデータで「既存戦略と統合すべき」と性急に統合する

**対策**: 判断には**最低30日、できれば60日**のデータが必要。短期のデータで判断しない。

### 9.2 テンプレートの頻繁な変更

- 毎日「ここも修正したい」と思ってテンプレートを変える
- 毎週のように新しいフィールドを追加する
- 改善のたびに過去データとの整合性が取れなくなる

**対策**: テンプレート変更は**月1回まで**。修正候補はメモに貯めてまとめて適用。

### 9.3 完璧主義

- 「全フィールドを埋めないと記録しない」
- 「時間がないから今日はスキップ」
- スキップが連続して運用停止

**対策**: **最重要3項目だけ**でも記録する。完璧な20日より、不完全な30日の方が価値がある。

### 9.4 改善なしでの継続

- 問題があっても「そのうち良くなる」と放置
- チェックリストを見ずに運用を続ける
- 数ヶ月後に「なぜ続けているのか」分からなくなる

**対策**: **週次チェックを必ず実施**。改善なき継続はプロジェクトを死なせる。

## 10. このドキュメント自体の改善

運用が進むと、この改善判断指示書自体にも改善余地が見えてきます：

- [ ] 「この基準は厳しすぎる/緩すぎる」と感じる
- [ ] 「こういう判断の場面が想定されていない」
- [ ] 「このフローチャートは実態に合わない」

**対策**: 
- 60日運用後、このドキュメントのバージョン2を作成
- 実際の改善履歴から、実効性のある基準に書き換える
- `improvement_log.md` を見直して、パターン化できる判断を追加

## 11. 最後に

このドキュメントは**不完全です**。運用してみないと分からない基準が多いからです。

最初の30日は、**このドキュメントを参考程度に**してください。致命的問題（セクション1）にだけ注意し、それ以外は運用を優先してください。

30日後、実際のデータを見ながらこのドキュメントを**あなた自身の基準**に書き換えてください。そこから本当の改善サイクルが始まります。

**改善の目的は「プロジェクトを続けること」です。** どんなに精密な基準を作っても、続かなければ意味がありません。まず続ける、次に改善する、の順を守ってください。

## 付録: 週次チェックの所要時間目安

- 致命的問題チェック（セクション1）: 5分
- データ量・品質確認（セクション2）: 10分
- テンプレート課題の認識（セクション3）: 10分
- シナリオ同質化の確認（セクション4）: 5分
- 既存戦略との相関確認（セクション5）: 10分
- 改善アクションの決定（セクション6〜7）: 10分

**合計: 約50分**

週1回、土曜または日曜の午後に時間を確保してください。この50分がプロジェクトの継続可否を決めます。
